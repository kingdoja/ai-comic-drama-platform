# V1 SCOPE

状态：Draft  
日期：2026-03-30  
适用版本：Product V1 定义冻结文档

## 1. 文档目的

这份文档用于冻结 AI 漫剧创作者工作台的 V1 产品边界。

它回答五个问题：

1. V1 到底服务谁
2. V1 卖的核心价值是什么
3. V1 必须交付哪些能力
4. V1 明确不做哪些能力
5. 后续新增需求如何判断是否应进入 V1

这份文档比 PRD 更强调“边界”，比 roadmap 更强调“取舍”。

---

## 2. V1 一句话定义

V1 是一个面向创作者和小团队的 AI 单集生产系统，帮助用户把女频网文章节或短剧脚本素材，稳定转成一个可预览、可审核、可返工、可导出的竖屏漫剧单集包。

---

## 3. V1 不是什麼

为了避免 scope 漂移，先明确 V1 不是以下产品：

1. 不是通用 AI 绘图工具
2. 不是通用脚本生成器
3. 不是全自动内容工厂
4. 不是多平台分发系统
5. 不是团队协同办公套件
6. 不是商业化投放分析平台
7. 不是 agent 自由协商的开放式多智能体系统

V1 的核心不是“生成很多东西”，而是：

> 稳定交付一个可发的单集包

---

## 4. 目标用户边界

## 4.1 核心目标用户

V1 只针对以下三类核心用户优化：

1. 女频网文改编团队
2. 竖屏短剧 / 漫剧内容工作室
3. 剧情类内容创作者或小型创作团队

## 4.2 次级用户

以下用户可以兼容，但不作为 V1 的主要优化对象：

1. MCN 内容运营
2. IP 孵化工作室
3. 教育剧情内容团队
4. 海外 webtoon 改编团队

## 4.3 明确不为其优化的用户

1. 需要多人实时协同编辑的大团队
2. 追求多平台一键分发的运营团队
3. 需要批量几十集自动生产的内容工厂
4. 需要复杂审核流、计费流和组织权限的企业客户

---

## 5. 用户购买的核心价值

用户买的不是“AI”，而是以下四个结果：

1. 更快：把原始素材更快转成一集可看的内容
2. 更稳：角色、剧情、镜头、资产尽量不跑偏
3. 更可控：关键节点可以编辑、锁定、审核
4. 更可修：出错后可以局部重跑，不必整条链重来

V1 任何功能，如果不能明显增强这四个价值之一，就不应该优先进入范围。

---

## 6. V1 最小交付单位

V1 的最小成功单位不是“项目建好了”，也不是“预览页能打开”，而是：

> 一个可交付的单集包

## 6.1 单集包定义

一个满足 V1 要求的单集包，至少包含以下产物：

1. Project Brief
2. Character Profile
3. Story Bible Summary
4. Episode Script
5. Storyboard / Shot List
6. Visual Spec
7. Keyframe Assets
8. Subtitle File
9. Voice Assets
10. Preview Video
11. QA Report
12. Final Export Bundle
13. Cover / Title / Publish Copy

## 6.2 单集包完成的最低标准

只有满足以下条件，才算 V1 的“完成交付”：

1. Workflow 主链路完整跑通
2. 关键文本产物可编辑、可锁定、可版本化
3. Preview 可播放
4. QA 报告已生成
5. 有人工审核门
6. 可以导出最终单集包
7. 失败可定位到具体 stage 或 asset

---

## 7. V1 产品范围总览

```text
输入素材
  -> Brief
  -> Story Bible / Character
  -> Script
  -> Storyboard
  -> Visual Spec
  -> Keyframe / Subtitle / TTS
  -> Preview
  -> QA
  -> Review Gate
  -> Final Export
```

V1 只覆盖这条主生产链及其必要管理能力。

---

## 8. V1 必做能力

## 8.1 项目与素材

### In Scope

1. 创建项目
2. 创建剧集
3. 粘贴文本素材
4. 上传基础素材文件
5. 设置目标平台
6. 设置目标时长
7. 设置项目基础元信息

### Out of Scope

1. 多来源复杂素材自动清洗
2. 批量导入多个项目
3. 外部平台自动抓取素材
4. 高级素材版权管理

### 验收线

1. 用户能独立创建项目和剧集
2. 素材进入系统后能被 workflow 消费
3. 项目基础信息可持久化

## 8.2 Brief 阶段

### In Scope

1. 生成项目 Brief
2. 展示题材、受众、卖点、风险
3. 支持编辑
4. 支持确认
5. 支持版本化

### Out of Scope

1. 自动 A/B 测试多个 brief 版本
2. 自动市场竞争分析
3. 自动题材评分系统

### 验收线

1. Brief 可生成
2. Brief 可编辑
3. Brief 可确认
4. Brief 确认后驱动下游 stage

## 8.3 Story Bible 与 Character

### In Scope

1. 生成 Story Bible Summary
2. 生成 Character Profile
3. 支持关键字段锁定
4. 支持编辑与版本化
5. 下游 stage 可读取这些约束

### Out of Scope

1. 高级关系图可视化编辑器
2. 完整百科式世界观库
3. 自动跨季人物管理

### 验收线

1. Character 和 Story Bible 都是独立可追踪产物
2. 锁定字段在下游默认只读
3. 修改后可形成新版本

## 8.4 Script

### In Scope

1. 生成单集剧本
2. 展示场次与对白
3. 支持编辑
4. 支持重生成
5. 支持版本化

### Out of Scope

1. 多集剧本自动联动生成
2. 完整长篇连续剧写作室能力
3. 剧情推荐市场

### 验收线

1. 剧本可消费为 storyboard 输入
2. 剧本重生成不会覆盖已确认旧版本
3. 剧本版本可追踪

## 8.5 Storyboard

### In Scope

1. 生成结构化分镜
2. 为每个镜头生成稳定 shot id
3. 编辑镜头信息
4. 锁定镜头
5. 按 shot 粒度支持后续重跑

### Out of Scope

1. 高级时间轴编辑器
2. 动画级镜头编排系统
3. 实时多人协作分镜工具

### 验收线

1. 每个 shot 都有唯一 id
2. 每个 shot 都有时长、角色、动作、视觉约束
3. shot 能作为 image_render 的独立输入

## 8.6 Visual Spec

### In Scope

1. 从 storyboard 生成视觉规格
2. 约束角色出镜、构图、服装、场景
3. 为 image_render 提供结构化输入

### Out of Scope

1. 完整风格市场
2. 自动风格迁移工作台
3. 高级图像美术控制面板

### 验收线

1. visual_spec 与 storyboard 分离
2. visual_spec 可独立重跑
3. image worker 可以直接消费 visual_spec

## 8.7 Keyframe / Image Render

### In Scope

1. 按 shot 渲染关键帧
2. 存储多候选资产
3. 选择主资产
4. 失败只重跑目标 shot
5. 记录基础 lineage

### Out of Scope

1. 复杂逐帧动画生成
2. 大规模批量画面工厂
3. 多供应商自动最优路由

### 验收线

1. 每个 shot 至少有一版可用图像结果
2. 资产入库并可被选中
3. 失败不会覆盖其他 shot 资产

## 8.8 Subtitle

### In Scope

1. 根据 script 与 shot 生成字幕稿
2. 导出字幕文件
3. 保持移动端阅读约束
4. 为 TTS 留出稳定切片边界

### Out of Scope

1. 多语言字幕系统
2. 平台自动适配多套字幕规范
3. 高级字幕动效编辑器

### 验收线

1. 字幕可导出
2. 单条字幕长度受控
3. 字幕与 shot、script 有稳定映射

## 8.9 TTS

### In Scope

1. 生成角色配音片段
2. 记录片段时长
3. 支持重跑目标片段
4. 产出音频资产

### Out of Scope

1. 高级配音导演控制台
2. 声线市场与版权分发
3. 多语言配音矩阵

### 验收线

1. 配音资产能用于 preview 合成
2. 片段时长可回写系统
3. 失败时可定位到具体片段

## 8.10 Preview Export

### In Scope

1. 合成 preview 视频
2. 生成基础封面
3. 生成基础标题和发布文案建议
4. 记录导出版本

### Out of Scope

1. 高级剪辑模板市场
2. 多平台一键导出多套格式
3. 云端批量渲染农场

### 验收线

1. preview 视频可播放
2. preview 结果进入资产库
3. 导出过程有明确成功或失败状态

## 8.11 QA

### In Scope

1. 生成 QA 报告
2. 展示问题清单
3. 记录严重级别
4. 给出 rerun stage 建议
5. 支持人工复核

### Out of Scope

1. 全自动替用户修复所有问题
2. 大规模离线评测平台
3. 复杂模型对战式质量系统

### 验收线

1. QA 输出不是单分数，而是结构化问题
2. 问题可映射到具体返工节点
3. QA 结果会影响是否允许导出

## 8.12 Human Review Gate

### In Scope

1. Brief 确认
2. Storyboard 确认
3. Final QA 审核
4. 审核通过 / 打回 / 拒绝

### Out of Scope

1. 多级审批流
2. 企业化会签流
3. 审批 SLA 和复杂通知系统

### 验收线

1. 审核结果写回 workflow
2. workflow 能被审核动作暂停或恢复
3. 未审核通过不得进入 final export

## 8.13 Final Export

### In Scope

1. 导出正式单集包
2. 打包关键产物
3. 保留导出记录

### Out of Scope

1. 自动投放
2. 自动发布到抖音 / 快手 / 小红书
3. 商业化数据归因系统

### 验收线

1. final export 必须依赖 review gate 通过
2. 导出包可复查
3. 导出结果有稳定记录

## 8.14 局部重跑能力

### In Scope

1. 从指定 stage 重跑
2. 对指定 shot 重跑 image_render
3. 对指定片段重跑 tts
4. 尽量复用非目标资产

### Out of Scope

1. 任意粒度无约束重跑
2. 自动推导所有最优返工路径
3. 跨项目复用复杂重跑树

### 验收线

1. rerun 不覆盖非目标对象
2. rerun 后 downstream 正确更新
3. rerun 结果和原版本都可追踪

## 8.15 可观测与运营基础

### In Scope

1. workflow 状态可见
2. stage 成功 / 失败可见
3. 基础日志
4. 基础 trace
5. 成本和耗时基础记录

### Out of Scope

1. 完整 BI 平台
2. 复杂告警中心
3. 高级报表中心

### 验收线

1. 任何失败都有落点
2. 能回答“卡在哪一步了”
3. 能回答“哪一步最慢、最贵、最容易错”

---

## 9. V1 明确不做的能力总表

### P1 后置

1. 更细的版本 diff 对比
2. 更智能的标题与封面建议
3. 项目模板和风格模板强化
4. 更丰富的 provider 适配层

### P2 后置

1. 批量多集自动生产
2. 团队协作
3. 数据复盘闭环
4. 多平台发布与投放分析
5. 高级模板市场

### 明确拒绝进入 V1

1. 自由协商式 multi-agent 对话系统
2. 为了“显得高级”而增加的复杂 orchestration
3. 没有明确交付价值的炫技功能
4. 会显著拖慢主链路的非必要可视化大工程

## 12. V1 上线门槛

V1 只有在以下条件同时满足时，才可视为达到目标：

1. 能创建项目并导入素材
2. 能走通 episode_workflow 主链路
3. 能生成并编辑关键文本产物
4. 能生成 preview
5. 能生成 QA 报告
6. 能通过 human review gate 控制导出
7. 能导出 final package
8. 能做 stage 级和 shot 级局部重跑
9. 能定位失败发生在哪个 stage
10. 至少有一条样板项目链路可重复成功运行

---

## 13. 新需求进入 V1 的判断规则

后续任何新需求，先回答下面四个问题：

1. 它是否直接增强“更快、更稳、更可控、更可修”中的一个
2. 它是否直接服务“单集包交付”这个最小成功单位
3. 它是否会显著拖慢主链路落地
4. 它是否可以安全后置到 P1 或 P2

### 进入 V1 的规则

只有同时满足以下条件，才可以考虑进入 V1：

1. 直接增强核心交付链
2. 不显著扩大系统复杂度
3. 不引入新的大类基础设施
4. 能在当前 roadmap 阶段内完成验收

### 不进入 V1 的规则

满足任一项就应该后置：

1. 更偏“锦上添花”而非“能否交付”
2. 更偏运营而非生产
3. 更偏规模化而非先跑通单集
4. 会明显增加 provider、权限、协同、平台集成复杂度

---

## 14. 最终冻结结论

V1 的边界可以浓缩为一句话：

> V1 只做一件事：把“单集包稳定做出来”这件事，做成一个可控、可审核、可返工、可导出的产品能力。

因此，任何不能直接帮助这个目标达成的能力，都不应优先进入 V1。
